- 是否足够支撑写作：足够。

## 关键事实

- Claude Code 内置 Learning output style，定位是 collaborative、learn-by-doing，不只是给解释，还会让用户自己补小块关键代码。
- Claude Code 的 output style 可以通过 `/config` 选择，设置会保存到项目本地配置；自定义 output style 也可以保存在用户级或项目级目录。
- Anthropic 研究中，52 名工程师学习陌生 Python 库 Trio，AI 组后续测验平均 50%，手写组 67%；AI 组完成任务只快了一点，但未达到统计显著。
- Anthropic 同一研究指出，AI 组内部差异来自使用方式：直接委托代码生成的人掌握更差，通过概念问题、解释和追问来使用 AI 的人保留更好。
- NotebookLM 的关键价值不是“凭空教你”，而是基于你导入的来源工作。官方文档说明，它会用来源里的直接引用、文本和图像作为引用，方便用户核对准确性。
- NotebookLM 支持 PDF、网页、YouTube 公开视频字幕、Google Docs/Slides、Markdown、音频等多种来源；但来源是导入时的静态副本，Google Drive 文件也需要手动同步。
- NotebookLM 可把来源转成思维导图、音频概览、视频概览、闪卡和测验；这些功能适合做学习输入、复习和自测，不适合替代真实编码练习。

## 反向数据 / 限制条件

- 不能把 Anthropic 研究外推成“所有 AI 编程辅助都会降低能力”。它研究的是 52 名相对初级工程师、陌生 Trio 库、特定实验任务。
- NotebookLM 不是编程执行环境，不能替你跑测试、调试本地项目或验证代码行为；它更适合做资料底座和复习系统。
- NotebookLM 来源可能是静态副本，网页、Drive 文档或官方文档更新后，学习资料需要重新同步或重新导入。
- NotebookLM 的 Audio / Video Overview、闪卡、测验仍可能有错误，官方帮助文档也提示需要核对。
- Claude Learning mode 只改变协作方式，不保证用户真的学习。如果用户跳过 `TODO(human)`、直接让 Claude 继续补完，本质上还是外包。

## 立骨

### 一句话主旨

用 AI 学编程，真正有效的方法不是“让它给答案”，而是搭一套训练系统：NotebookLM 负责把资料变成可检索的知识底座，Claude Learning mode 负责把练习拆成你必须亲手完成的小步，真实项目、测试和复盘负责验收。

### 结构

1. 开场：很多人用 AI 学编程，学到最后只是复制得更熟。
2. 判断：AI 默认优化交付，不默认优化能力形成。
3. 证据：Anthropic 编程学习研究说明“怎么用 AI”比“用不用 AI”更关键。
4. 系统：三层训练系统。
   - NotebookLM：资料层，解决“我学什么、从哪里学、如何复习”。
   - Claude Learning：练习层，解决“我怎么动手、哪里要自己写”。
   - 真实项目：验收层，解决“我到底会不会用”。
5. SOP：7 天入门一个编程主题的流程。
6. 模板：NotebookLM prompt、Claude prompt、复盘模板。
7. Human3.0 收束：AI 是脚手架，不是替身；能力必须长在自己身上。

## 公众号正文

现在用 AI 学编程，太容易了。

你想学 Python，让它写一份路线图。你想学 React，让它解释组件。你遇到报错，把终端日志粘进去。你想做一个小工具，让它直接生成代码。

几分钟后，答案出来了。代码也能跑，页面也能显示，报错也消失了。

问题是：你真的学会了吗？

很多人用 AI 学编程，最后只是变成了“更熟练地复制答案”。他们会问，会粘，会让 AI 改错，也能在短时间内做出一点东西。但一旦离开 AI，或者项目换一个环境、报错换一种形态、需求变复杂一点，就不知道该从哪里下手。

这不是因为 AI 不适合学编程。

恰恰相反，AI 可能是普通人学习编程以来最强的辅助工具。它能解释概念，拆解代码，模拟老师，陪你调试，给你出题，还能帮你把零散资料整理成学习系统。

真正的问题是：你把 AI 放在了什么位置。

如果你把它当答案机器，它会让你更快完成任务，但不一定让你长出能力。

如果你把它当训练系统，它可以把原来很难坚持的编程学习，拆成资料、练习、反馈、验收和复盘。

这两种用法，结果完全不同。

### 1. 学编程最怕的不是没有答案，而是没有手感

编程不是背语法。

语法当然要懂，但真正让一个人从“看得懂一点”进入“能自己做东西”的，是另一组能力：

- 能读懂一段陌生代码在做什么。
- 能判断一个报错可能从哪里来。
- 能把一个需求拆成几个可验证的小步骤。
- 能知道 AI 生成的代码哪里可能不靠谱。
- 能把这次踩坑沉淀成下一次可以复用的经验。

这些能力都需要手感。

手感不是看答案看出来的。手感来自你自己改过代码、跑过测试、撞过错误、解释过原因。

这也是为什么“直接让 AI 写完整答案”很危险。它短期看起来省时间，长期却可能让你跳过最关键的训练动作。

Anthropic 在 2026 年发布过一项关于 AI 编程辅助和技能形成的研究。研究让 52 名工程师学习一个陌生的 Python 异步库 Trio，一组可以使用 AI，一组手写完成。结果很值得警惕：AI 组后续理解测验平均 50%，手写组 67%；AI 组完成任务只快了一点，但没有达到统计显著。

更关键的是，AI 组内部差异很大。

那些直接让 AI 生成代码、替自己完成任务的人，掌握度更差。那些会向 AI 提概念问题、要求解释、追问为什么的人，学习效果更好。

这说明问题不在“能不能用 AI”，而在“你有没有把自己留在学习回路里”。

### 2. 我建议把 AI 编程学习拆成三层

第一层，资料层。

你需要一个地方放官方文档、教程、课程笔记、报错记录、项目 README、自己的复盘。这个地方不是普通聊天框，因为聊天框很容易散，也很难长期复用。

这里适合用 NotebookLM。

NotebookLM 的优势是基于你上传的来源工作。你可以把官方文档、PDF、网页、YouTube 视频字幕、Markdown 笔记、课程资料放进去，让它基于这些资料回答问题，并带引用。它还可以生成思维导图、学习指南、音频概览、视频概览、闪卡和测验。

对学编程来说，它最适合解决三个问题：

我到底该学哪些概念？这些概念之间是什么关系？我怎么定期复习和自测？

第二层，练习层。

你需要一个能陪你写代码、但不会直接把所有东西替你写完的工具。

这里适合用 Claude Code 的 Learning output style。

Claude 官方文档里，Learning output style 的定位是“边做边学”。它不只是给你解释，还会在代码里留下 `TODO(human)`，要求你自己完成小块关键代码。

这个设计很重要。因为学编程最容易被外包掉的，就是那一点点“我自己写”的动作。

你不需要一上来就独立完成整个项目，但你必须在关键位置动手。哪怕只写一个函数、补一个判断、修一个测试、解释一个错误，都是在把能力留在自己身上。

第三层，验收层。

你需要一个真实项目、一个测试、一段可运行代码，来证明自己不是“看懂了答案”，而是真的能把知识用出来。

NotebookLM 不能替你跑代码。Claude 的解释也不能替代项目结果。

真正的验收很朴素：功能能不能跑，测试能不能过，报错能不能定位，换一个类似需求你能不能自己改。

这三层合起来，才是一套比较完整的 AI 编程训练系统：

NotebookLM 管资料。

Claude Learning 管练习。

真实项目和测试管验收。

### 3. NotebookLM 怎么用：先搭一个“知识底座”

假设你想学 React。

很多人的第一步是问 AI：“给我一份 React 学习路线。”

这一步可以做，但不够。

更好的做法是先建一个 NotebookLM notebook，把学习资料放进去：

- React 官方文档里的核心章节。
- 你正在看的课程链接或视频字幕。
- 你自己的学习笔记。
- 你做项目时遇到的报错记录。
- 你写过的小 demo 的 README。

然后不要问它“帮我总结 React”。

这个问题太大，得到的往往是一篇正确但没用的概述。

你可以这样问：

```text
请基于我上传的资料，整理 React 入门最核心的 10 个概念。
每个概念请说明：
1. 它解决什么问题；
2. 初学者最容易误解什么；
3. 我应该用什么小练习验证自己真的懂了；
4. 引用来源位置。
```

接着再问：

```text
请把这些概念组织成一张学习路径：
先学什么，后学什么，哪些可以暂时跳过。
不要追求全面，目标是让我能做出一个可运行的小项目。
```

然后让 NotebookLM 生成闪卡和测验。

注意，这里不是为了考试，而是为了防止自己“以为懂了”。编程学习里最常见的幻觉不是 AI 的幻觉，而是人的幻觉：看解释时觉得懂，自己写时全忘了。

闪卡、测验、思维导图、音频概览，适合做输入和复习。它们不负责让你成为程序员，但可以让你的资料不再散。

这就是第一层：把学习资料变成可检索、可复习、可引用的知识底座。

### 4. Claude Learning mode 怎么用：让 AI 留作业，而不是替你交作业

资料整理完之后，下一步不是继续看。

下一步是写代码。

如果你用 Claude Code，可以选择 Learning output style。官方说明里，这个模式会在编码时给出理解提示，并要求你自己贡献小块关键代码。

你可以这样开一个学习任务：

```text
我正在学习 React state 和 effect。
请用 Learning output style 带我做一个最小练习。
要求：
1. 不要直接给完整答案；
2. 先解释这个练习训练什么能力；
3. 把代码拆成 3-5 个小步骤；
4. 至少留下 2 个 TODO(human)，让我自己完成；
5. 每一步结束后问我一个检查理解的问题；
6. 最后用测试或手动验收标准确认我真的会了。
```

这段 prompt 的关键，不是让 Claude 更聪明，而是给自己设边界。

你是在告诉 AI：不要替我把练习吃掉。

学编程时，最有价值的不是 Claude 给你的完整代码，而是它逼你完成的那几个小动作：

- 你自己写一个条件判断。
- 你自己解释一段错误。
- 你自己补一个测试。
- 你自己把一个函数拆小。
- 你自己说出为什么这样写。

这些动作看起来慢，但它们才是能力形成的地方。

### 5. 一套 7 天 AI 编程训练流程

如果你想真正开始，我建议不要先立一个“我要学会编程”的大目标。

目标太大，很容易变成收藏路线图。

选一个很小的主题。

比如：

- 用 Python 写一个文件整理脚本。
- 用 React 做一个待办清单。
- 用 Node.js 写一个简单接口。
- 用 SQL 查询一份表格数据。
- 用 Shell 自动化一个重复动作。

然后按 7 天跑一轮。

第 1 天：建资料底座。

把官方文档、一个教程、一个示例项目、自己的问题清单放进 NotebookLM。让它整理概念地图和最小学习路径。

第 2 天：做第一轮概念自测。

让 NotebookLM 生成 10 张闪卡和 5 道测验。错题不要删，直接变成“我不懂清单”。

第 3 天：用 Claude 做第一个最小练习。

要求 Claude 使用 Learning output style，不给完整答案，必须留下 `TODO(human)`。你的任务是完成这些小块代码，而不是复制整段结果。

第 4 天：故意制造一个错误。

让 Claude 先不要修，只让它带你读报错、定位范围、列可能原因。你自己先写判断，再让它校准。

第 5 天：做一个微型项目。

项目越小越好，但必须完整。比如一个命令行工具、一个页面、一个 API。目标是跑通，而不是炫技。

第 6 天：让 AI 做 code review。

不要问“有没有问题”。要问：

```text
请从初学者能力训练角度 review 这段代码：
1. 哪些地方我可能只是照抄，没有真正理解；
2. 哪些 bug 风险值得我自己先排查；
3. 哪 3 个问题最适合拿来复习；
4. 请不要直接重写，先给我检查路径。
```

第 7 天：沉淀成资产。

写一页复盘：

- 我学会了什么概念。
- 我踩了什么坑。
- 我下次遇到类似问题先看哪里。
- 我可以沉淀成什么 checklist、模板或 prompt。

这一步最容易被省略，但它决定你是在“用 AI 完成一次任务”，还是在“把一次任务变成自己的数字生产资料”。

### 6. 编程学习里，AI 最适合扮演四个角色

第一个角色：资料整理员。

NotebookLM 很适合做这个。你把资料放进去，它帮你提炼概念、组织路径、生成复习材料。但资料来源要你选，引用要你核对，更新也要你维护。

第二个角色：概念教练。

当你读不懂闭包、异步、状态管理、数据库索引时，AI 可以换三种说法解释，也可以给生活类比。但你不能只听解释，你要让它出题，逼自己复述。

第三个角色：练习搭档。

Claude Learning mode 的价值就在这里。它可以把一个练习拆小，把最关键的部分留给你，让你在低压力环境里动手。

第四个角色：复盘助手。

每次做完一个小项目，让 AI 帮你从错误、概念、可复用模板三个角度复盘。复盘不是写心得，而是沉淀下次可以直接复用的结构。

如果你只让 AI 当第一个和第二个角色，你会看起来学了很多。

如果你能让 AI 当第三个和第四个角色，你才更可能真的长能力。

### 7. 最重要的规则：先让自己进入问题

用 AI 学编程，有一个简单规则：

凡是会影响能力形成的任务，先让自己进入问题，再让 AI 介入。

不要一上来就问：

```text
帮我写一个 React todo 应用。
```

换成：

```text
我想做一个 React todo 应用。
我先猜核心步骤是：组件拆分、状态管理、事件处理、本地存储。
请先判断我的拆解是否合理，然后给我第一个最小练习。
不要给完整代码，保留 TODO(human) 让我写关键部分。
```

不要一上来就问：

```text
这个报错怎么修？
```

换成：

```text
我怀疑这个报错和异步调用顺序有关，也可能是类型不匹配。
请先帮我沿调用链定位最可能的 3 个原因。
先不要改代码，给我最小验证步骤。
```

不要一上来就问：

```text
帮我总结这篇文档。
```

换成：

```text
请基于这篇文档，列出我做一个最小项目必须掌握的概念。
每个概念给一个小练习，并生成 5 道自测题。
```

同样是用 AI，差别很大。

前者是索取答案。

后者是组织训练。

### 8. Human3.0 的核心，不是不用 AI，而是不交出判断权

我一直觉得，AI 时代真正重要的不是“会不会用某个工具”，而是你能不能把工具组织成自己的系统。

学编程也是一样。

如果你只是不断问 AI 要答案，你会越来越依赖它。

如果你用 NotebookLM 建资料底座，用 Claude Learning mode 做练习，用真实项目和测试做验收，再把每次踩坑沉淀成 checklist、SOP、prompt、skill，你就不是在消费 AI，而是在建设自己的生产系统。

AI 可以替你查语法，解释概念，生成示例，指出错误。

但它不能替你长出编程手感。

它不能替你知道自己到底懂不懂。

它不能替你把一次错误变成下次的判断力。

所以，用 AI 学编程最重要的一句话是：

别让 AI 替你完成学习。

让 AI 训练你完成学习。

## 知乎版本

### 推荐问题标题

如何高效利用 AI 学习并真正掌握编程，而不是只会复制代码？

### 回答

我现在对“用 AI 学编程”的判断很明确：不要把 AI 当答案机器，要把 AI 搭成训练系统。

很多人用 AI 学编程的方式是错位的。遇到问题直接问“帮我写”“帮我修”“帮我解释”。短期看效率很高，长期很容易出现一个问题：项目能跑，但自己没有形成编程手感。

编程能力不是由“看过多少答案”构成的，而是由几类动作构成的：读代码、拆需求、定位错误、选择方案、验证结果、复盘迁移。

这些动作如果全部交给 AI，人就只剩下复制和运行。

更好的方式是三层系统。

第一层，用 NotebookLM 做知识底座。

把官方文档、教程、项目 README、自己的笔记、报错记录、课程资料放进去。NotebookLM 的价值是基于你上传的来源工作，可以带引用回答问题，也可以生成学习指南、思维导图、音频概览、闪卡和测验。

不要只问“帮我总结这个框架”。更好的问法是：

```text
基于我上传的资料，整理这个框架入门必须掌握的 10 个概念。
每个概念说明它解决什么问题、初学者容易误解什么、可以用什么小练习验证。
```

第二层，用 Claude Code Learning output style 做练习。

Claude 的 Learning output style 不是单纯解释模式。官方文档里说，它会在编码过程中给出理解提示，也会让用户自己完成小块关键代码，比如留下 `TODO(human)`。

这个机制对学编程很重要。你不需要一开始独立写完整项目，但必须亲手完成关键小块。否则你只是看懂了 AI 的答案，没有把能力长到自己身上。

我会这样要求 Claude：

```text
请用 Learning output style 带我做一个最小练习。
不要直接给完整答案。
请解释这个练习训练什么能力，把任务拆成 3-5 步，并至少留下 2 个 TODO(human) 让我自己完成。
每一步后问我一个理解检查问题。
```

第三层，用真实项目和测试做验收。

NotebookLM 不能替你跑代码，Claude 的解释也不能证明你会了。真正的验收是：功能能不能跑，测试能不能过，换一个类似需求你能不能自己改。

我建议初学者不要定“学会 Python / 学会 React”这种大目标。选一个小到不能再小的项目：文件整理脚本、todo 页面、简单 API、SQL 查询练习。每轮只学一个概念，做一个练习，留一次复盘。

一套 7 天流程可以这样跑：

第 1 天，用 NotebookLM 建资料底座。  
第 2 天，让 NotebookLM 生成概念地图、闪卡和测验。  
第 3 天，用 Claude Learning 做第一个最小练习。  
第 4 天，故意制造一个错误，训练调试。  
第 5 天，做一个微型项目。  
第 6 天，让 AI 做 code review，但不直接重写。  
第 7 天，把错误、概念和 prompt 沉淀成复盘文档。

这里最关键的是第 7 天。很多人用 AI 学编程没有沉淀，所有经验都留在聊天记录里。下一次遇到类似问题，又重新问一遍。

真正的 Human3.0 用法，是把每次学习变成自己的数字生产资料。一个错题清单、一个调试 checklist、一个项目模板、一个 prompt 库，都会让你下次更快、更稳。

所以我不建议初学者拒绝 AI。

我建议反过来：更深地使用 AI，但要把自己的判断、动手和验收留在流程里。

AI 可以帮你解释、拆解、反馈、陪练。

但它不能替你长出编程能力。

## Prompt 模板库

### NotebookLM：建立资料底座

```text
请基于我上传的资料，整理这个编程主题的最小学习路径。
要求：
1. 只保留做出一个最小项目必须掌握的概念；
2. 每个概念说明它解决什么问题；
3. 标出初学者最容易误解的地方；
4. 给每个概念设计一个 15 分钟小练习；
5. 所有结论都附来源引用。
```

### NotebookLM：生成自测

```text
请基于当前资料生成一组自测题：
1. 5 道概念题；
2. 5 道读代码题；
3. 3 道调试题；
4. 每道题给提示，但先不要给答案；
5. 最后输出我应该重点复习的薄弱概念。
```

### Claude Learning：做最小练习

```text
请用 Learning output style 带我完成这个练习。
目标：让我真正掌握 <概念名>。
规则：
1. 不要直接给完整代码；
2. 先解释这个练习训练什么能力；
3. 把任务拆成 3-5 个小步骤；
4. 至少保留 2 个 TODO(human) 让我自己写；
5. 我写完后，你再 review；
6. 最后用测试或手动验收标准确认我真的会了。
```

### Claude：调试训练

```text
下面是报错和相关代码。
请先不要直接修。
请按这个顺序带我排查：
1. 报错说明了什么；
2. 最可能的 3 个原因；
3. 每个原因的最小验证步骤；
4. 让我先选择一个方向并说明理由；
5. 我验证后，你再给下一步建议。
```
