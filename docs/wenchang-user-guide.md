# 文昌.skill 用户使用指南

这份指南用于真实跑一遍文昌内容流水线。默认使用“总控模式”：用户提出目标后，文昌总控自动选择子路径并推进，遇到需要人判断的节点再停下来提问。

推荐先用一个真实主题跑完整链路，再考虑拆小红书、知乎或归档。

## 推荐用法：总控模式

你可以直接这样开始：

```text
从这个主题开始，走完整套文昌流程，遇到需要我判断的节点再停下来问我。

主题：我以为 Codex 是写代码的，结果它开始接管我的工作台
素材：https://jxnl.github.io/blog/writing/2026/05/10/codex-maxxing/
账号：AI生命克劳德
长期方向：Human3.0
目标平台：公众号
要求：不要翻译原文，要写成我自己的判断文。
```

总控会先判断入口阶段，再选择子路径。你不需要手动复制每一步输出。

## 适合输入的三类任务

### 1. 已有方向，但没有初稿

适合从路由和采证开始。

```text
帮我把“AI 记忆能力会改变普通人的工作方式吗”这个方向做成一篇公众号文章。

账号：AI生命克劳德
长期方向：Human3.0
目标读者：普通 AI 使用者、创作者、工程师
当前没有初稿
```

### 2. 外部文章 / 热点素材

适合先判断能不能转成自己的选题，不要直接复述原文。

```text
把这篇文章当成热点素材，判断是否适合写公众号。

素材链接：<URL>
账号：AI生命克劳德
长期方向：Human3.0
目标：不要翻译原文，要提炼成我自己的选题。
```

### 3. 已有初稿

适合先诊文，不要直接重写。

```text
请把下面这篇初稿用文昌·诊文检查一遍，先判断是否值得继续写，再决定要不要重构。

目标平台：公众号
账号：AI生命克劳德
长期方向：Human3.0

<粘贴初稿>
```

## 一次完整流程

```text
探脉 -> 定题 -> 采证 -> 立骨 -> 起稿 -> 诊文 -> 整章 -> 出刊 -> 配图/卡片/上传 -> 归档
```

真实使用时不一定从探脉开始。总控会根据输入选择子路径。

## 入口对应子路径

| 输入状态 | 默认子路径 |
| --- | --- |
| 不知道写什么 | 探脉 -> 定题 -> 采证 -> 立骨 -> 起稿 -> 诊文 -> 整章 -> 出刊 -> 配图/卡片/上传 -> 归档 |
| 已有明确主题 | 定题 -> 采证 -> 立骨 -> 起稿 -> 诊文 -> 整章 -> 出刊 -> 配图/卡片/上传 -> 归档 |
| 外部文章/热点素材 | 定题 -> 采证 -> 立骨 -> 起稿 -> 诊文 -> 整章 -> 出刊 -> 配图/卡片/上传 -> 归档 |
| 已有初稿 | 诊文 -> 采证/整章/重写 -> 出刊 -> 配图/卡片/上传 -> 归档 |
| 只要求发布检查 | 出刊 -> 配图/卡片/上传 -> 归档 |

## 总控会自动停下来的情况

- 选题有多个可行方向，需要你选一个。
- 采证可信度低，或缺少反向证据。
- 诊文建议重写、暂不投入或转平台。
- 整章编辑需要大幅删改。
- 出刊缺标题、摘要、封面、标签、配图。
- 需要生成配图、卡片、上传外部图片。
- 建议进入 Human3.0 成书审查或素材库。
- 你明确要求先停。

## 手动模式

下面的分步流程用于调试、学习和回归。如果你使用总控模式，不需要每一步复制粘贴。

## 第 1 步：路由 / 定题

目标：确认平台、阶段、选题角度和下一步。

你可以这样说：

```text
用文昌路由处理这个需求：

<粘贴主题、链接或初稿>
```

期望输出：

- `## 路由判断`
- `## brief`
- `content_state`
- `handoff`
- 下一步应该调用哪个 skill

路由阶段只应该给 brief，不应该直接写正文。

brief 至少包括：

- `Angle`：核心切口
- `Hook`：第一句候选，必须是判断句
- `Subpoints`：3 个分论点
- `What to avoid`：不该写的角度
- `Suggested format`：平台、篇幅、形态

如果这一步已经开始写完整文章，说明流程跑偏了，要退回路由。

## 第 2 步：采证

目标：为选题补事实、来源、反向证据和可信度。

把上一步的 `content_state` 和 `handoff` 交给采证：

```text
请用文昌·采证处理下面的 content_state 和 handoff。

<粘贴上一步输出>
```

期望输出：

- `## 采证结论`
- `## 来源清单`
- `## 关键事实`
- `## 反向数据 / 限制条件`
- `## 可引用句子`
- `content_state.research`
- `handoff`

采证阶段最重要的是 `contrarian_points`。如果没有反向证据，就要降低可信度，并说明还需要验证什么。

不要接受没有来源的确定性数据。

## 第 3 步：起稿

目标：基于 brief 和 research 写完整初稿。

```text
请用 AI / Human3.0 公众号写作 skill 起稿。

只读取下面这些内容：
- content_state.topic
- content_state.audience
- content_state.research
- handoff.accepted_inputs

不要重新选题，不要新增未采证的强事实。

<粘贴上一步输出>
```

期望输出：

- 标题
- 正文初稿
- `content_state.draft`
- 下一步指向 `wenchang-review`
- `handoff`

起稿阶段可以写得完整，但不做最终发布判断。

## 第 4 步：诊文

目标：判断初稿是否值得继续推进。

```text
请用文昌·诊文检查这篇初稿。

目标平台：公众号
请先诊断，不要默认全文重写。

<粘贴初稿和 content_state>
```

期望输出：

- `## 诊断结论`
- `## 核心问题`
- `## 最小修改建议`
- `## 平台适配`
- `## 长期资产判断`
- `content_state.diagnosis`
- `handoff`

如果结论是“整章编辑”，再进入下一步。

## 第 5 步：整章编辑

目标：把可用初稿改成发布前正文。

```text
请用文昌·诊文的整章模式处理这篇稿子。

要求：
- 删除 20%-30% 的空泛、重复、无证据内容
- 保留 Human3.0 主线
- 每个关键判断尽量接事实、案例或作者经验
- 最后一句要能独立传播

<粘贴初稿、诊断结论和 research>
```

期望输出：

- `## 编辑结论`
- `## 编辑后正文`
- `## 修改日志`
- `## 删减说明`
- `## 最强一句`
- `content_state`
- `handoff`

整章模式不是润色。它应该删、重排、压缩和强化结尾。

## 第 6 步：出刊

目标：补齐发布资产，不再重写正文。

```text
请用文昌·出刊检查这篇编辑后正文。

目标平台：公众号
如果缺标题、摘要、封面、标签、转发文案，请列为必补项。

<粘贴编辑后正文和 content_state>
```

期望输出：

- `## 发布结论`
- `## 必补项`
- `## 平台发布包`
- `## 归档建议`
- `content_state.publish_assets`
- `content_state.distribution`
- `content_state.archive`
- `handoff`

出刊阶段不应该偷偷改正文。如果正文质量有问题，要退回诊文或整章。

## 第 7 步：配图 / 卡片

如果要做小红书或朋友圈二次分发，可以继续说：

```text
基于这篇发布版文章，生成小红书 7 页卡片结构和封面文案。

要求：
- 每页只讲一个点
- 保留 Human3.0 主线
- 不新增正文没有的事实
```

如果 Markdown 里有本地图片，发布前再走 `md-img-r2`：

```text
用 md-img-r2 上传这篇 Markdown 里的本地图片，并替换成公开 URL。

文件：<Markdown 路径>
```

## 第 8 步：归档

如果文章适合长期沉淀：

```text
请用 Human3.0 成书守门员审查这篇文章，判断是否适合进入素材库或成书结构。

<粘贴发布版正文和 content_state.archive>
```

归档不是自动入书。它只给判断和建议，最后由人决定。

## 每一步都要保留的东西

每一步结束后，至少保留：

- 当前正文或阶段输出
- `content_state`
- `handoff`
- 用户确认或修改意见

这三样是文昌流水线能持续接力的基础。

## 常见跑偏信号

| 信号 | 处理方式 |
| --- | --- |
| 路由阶段直接写正文 | 退回路由，只要 brief |
| 采证没有反向数据 | 要求补 `contrarian_points` |
| 写作阶段重新选题 | 要求只使用 `topic`、`research` 和 `handoff` |
| 诊文只夸不指出问题 | 要求给核心问题和最小修改建议 |
| 整章只是润色 | 要求删减 20%-30%，输出删减说明 |
| 出刊阶段改正文 | 退回诊文或整章 |
| 发布前没有阻塞项 | 检查标题、摘要、封面、标签、配图、转发文案 |

建议先跑这个：

```text

```

输入可以这样写：

```text
我想用文昌完整跑一篇公众号文章。

主题：我以为 Codex 是写代码的，结果它开始接管我的工作台
素材：https://jxnl.github.io/blog/writing/2026/05/10/codex-maxxing/

请先从文昌路由开始，只输出 brief、content_state 和 handoff，不要写正文。
```

## 完成标准

一轮完整流程结束时，应至少得到：

- 一篇公众号发布前正文
- 标题、摘要、标签、封面文案
- 朋友圈转发文案
- 评论区引导
- 小红书/知乎二次分发建议
- 是否进入 Human3.0 素材库的判断
- 一份可复用的 `content_state`
