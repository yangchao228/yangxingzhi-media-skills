# 文昌.skill

> 从热点到发布，把自媒体写作全流程收拢成一套可复用的内容生产系统。

`文昌.skill` 是一套面向自媒体创作者的写作技能体系。它覆盖热点探测、选题、提纲、初稿、插图、诊断、排版、封面、发布等主要环节，但不要求用户自己判断“下一步该调用哪个技能”。

所以这套体系必须有一个总调度入口：

- 用户只说目标
- 路由 skill 判断当前处于哪个创作阶段
- 决定调用一个子技能，还是按顺序串起多个子技能
- 在全流程里保持选题方向、语气、结构和平台风格一致

---

## 命名结构

整套体系建议这样命名：

- 系统总名：`文昌.skill`
- 总调度：`文昌路由.skill`
- 子技能：
  - `文昌·探脉`
  - `文昌·定题`
  - `文昌·采证`
  - `文昌·立骨`
  - `文昌·起稿`
  - `文昌·配图`
  - `文昌·诊文`
  - `文昌·整章`
  - `文昌·封面`
  - `文昌·出刊`

这里的关系不是并列插件堆叠，而是：

- `文昌.skill` 是总品牌
- `文昌路由.skill` 是主入口
- 其他子技能是具体执行模块

## 第一版工作流落地

当前仓库已经把现有内容创作 skills 归一到 `content/` 目录，并补齐核心工作流入口：

- `content/wenchang-orchestrator/`：总控，选择完整子路径、自动推进可自动阶段、在人工判断节点暂停
- `content/wenchang-router/`：总调度，识别平台和阶段
- `content/wenchang-research/`：采证，补来源、事实、反向数据和可信度
- `content/wenchang-review/`：诊文，判断初稿是否值得继续推进
- `content/wenchang-publish-check/`：出刊，做发布前检查

真实使用时先看 [文昌.skill 用户使用指南](docs/wenchang-user-guide.md)。默认用文昌总控自动推进；手动模式只用于调试和回归。

核心入口围绕统一的 `content_state` 接力，字段规范见 `content/CONTENT_STATE.md`。每个核心入口都带有最小回归样例：

- `examples/minimal-input.md`
- `examples/expected-output-notes.md`

当前实现是一个 agent 调度多个 skill，但按伪多 agent 方式运行：每个阶段只读取 `content_state`、上一阶段正式输出和用户明确新增材料，不沿用未结构化聊天历史。跨阶段交接统一写入 `handoff`，后续可以平滑拆成真正多 agent。

## 为什么要有总调度

如果没有总调度，用户就得自己判断：

- 现在应该先找热点，还是先定题
- 是先写提纲，还是直接起稿
- 是先诊断，还是先排版
- 配图、封面、发布应该何时介入

这会让整个系统很快退化成“很多技能，但不会用”。

所以 `文昌路由.skill` 的职责不是简单转发，而是承担三件事：

- 理解用户当前目标
- 判断最合理的下一步
- 把多个技能串成一条可执行的内容生产链

也就是说，用户理想上只需要说：

- `帮我找 3 个最近能写的 AI 选题`
- `把这个主题做成公众号文章`
- `这个选题先帮我采证，再决定怎么写`
- `这篇稿子先诊断，再排版成发布版`
- `给这篇文章补插图和封面方案`

剩下的阶段判断、技能切换和结果衔接，交给 `文昌路由.skill` 处理。

## 总调度技能

### `文昌路由.skill`

用途：

- 接收用户自然语言需求
- 判断当前处于哪一个创作阶段
- 调度合适的单个子技能，或按顺序调用多个子技能
- 保持整篇内容在主题、风格、结构上的一致性
- 在关键节点提醒用户保留人工判断权

它负责的典型任务：

- 从热点池里选一个最值得写的题
- 从题目一路推进到提纲、初稿和发布版
- 对已有稿件做诊断、改写、排版和封面补齐
- 根据不同平台把同一主题改造成不同版本

它不应该做的事：

- 直接吞掉所有细分职责，变成一个什么都写的万能 prompt
- 跳过中间阶段，输出无法复用的“一次性结果”
- 在用户没有确认关键方向时，替代用户做不可逆的最终判断

推荐触发语：

- `帮我做一篇关于 XXX 的公众号文章`
- `最近这个方向有什么值得写的`
- `从选题到发布，帮我走完整套流程`
- `先帮我诊断这篇，再决定要不要重写`

## 子技能结构

### `文昌·探脉`

用途：

- 热点探测
- 素材扫描
- 趋势初筛

输入：

- 主题领域
- 关注平台
- 时间范围
- 目标受众

输出：

- 热点列表
- 热度与时效判断
- 可写性判断
- 值得继续深入的候选方向

适用场景：

- 用户不知道最近写什么
- 用户想围绕一个方向找切口
- 用户需要先建立选题池

不负责：

- 最终拍板选题
- 直接产出完整文章

### `文昌·定题`

用途：

- 选题筛选
- 角度确定
- 标题方向设计

输入：

- 热点池
- 账号定位
- 读者画像
- 内容目标

输出：

- 选题优先级排序
- 推荐题目
- 写作切口
- 标题方向

适用场景：

- 已经有多个候选题，但不知道先写哪个
- 需要从热点变成“可写、可发、可读”的正式选题

不负责：

- 文章结构展开
- 全文生成

### `文昌·立骨`

用途：

- 提纲设计
- 结构拆解
- 论点排序

输入：

- 已确认选题
- 平台类型
- 篇幅要求
- 内容目标

输出：

- 文章结构
- 段落分配
- 核心论点
- 案例与素材插入位置

适用场景：

- 题目已定，但文章还没有骨架
- 需要先把逻辑打稳，再写初稿

不负责：

- 直接写完整终稿

### `文昌·起稿`

用途：

- 初稿生成
- 段落扩写
- 从提纲到正文

输入：

- 提纲
- 写作语气
- 平台类型
- 目标字数

输出：

- 可继续修改的完整初稿

适用场景：

- 需要快速产出第一版正文
- 已有提纲或结构明确

不负责：

- 最终定稿
- 封面、排版、发布收尾

### `文昌·配图`

用途：

- 插图方案设计
- 视觉节奏规划
- 图片 prompt 生成

输入：

- 文章内容
- 平台类型
- 视觉风格
- 配图数量要求

输出：

- 配图建议
- 图文对应关系
- 图片提示词
- 视觉风格说明

适用场景：

- 文章需要增强可读性
- 需要交给画图模型或设计工具继续执行

不负责：

- 最终版封面标题策略

### `文昌·诊文`

用途：

- 内容诊断
- 结构问题识别
- 表达质量评估

输入：

- 草稿或成稿
- 平台目标
- 写作目标

输出：

- 问题清单
- 修改建议
- 优先级排序
- 是否建议重写的判断

适用场景：

- 文章写出来了，但不确定是否能发
- 想知道问题出在结构、表达还是信息密度

不负责：

- 直接生成排版版式

### `文昌·整章`

用途：

- 排版整理
- 平台适配
- 发布版文本整理

输入：

- 成稿
- 目标平台
- 风格要求

输出：

- 适合目标平台的排版版本
- 段落节奏优化后的正文
- 小标题、分隔、强调结构

适用场景：

- 同一篇稿子要适配公众号、知乎、小红书等不同平台
- 需要把普通正文整理成可发布版本

不负责：

- 热点判断
- 选题判断

### `文昌·封面`

用途：

- 封面标题
- 封面文案
- 主视觉方向

输入：

- 文章主题
- 平台类型
- 内容调性

输出：

- 封面标题方案
- 副标题或辅助文案
- 封面视觉提示词
- 版式方向建议

适用场景：

- 发布前补齐点击入口
- 需要统一标题策略与视觉表达

不负责：

- 正文改写

### `文昌·出刊`

用途：

- 发布前收尾
- 多平台发布准备
- 输出最终交付物

输入：

- 最终稿
- 平台目标
- 发布时间

输出：

- 发布版正文
- 摘要
- 标签
- 发布说明
- 平台备注

适用场景：

- 文章已经完成，需要进入实际发布环节
- 要做公众号、知乎、小红书等平台的最后整理

不负责：

- 前置热点搜索
- 选题筛选

## 技能调用链路

最常见的调用链路有这几种：

### 1. 从零开始写

`文昌路由.skill`
-> `文昌·探脉`
-> `文昌·定题`
-> `文昌·立骨`
-> `文昌·起稿`
-> `文昌·诊文`
-> `文昌·整章`
-> `文昌·封面`
-> `文昌·出刊`

适用：

- 用户只有一个方向，没有现成稿件

### 2. 已有题目，直接成文

`文昌路由.skill`
-> `文昌·立骨`
-> `文昌·起稿`
-> `文昌·诊文`
-> `文昌·整章`

适用：

- 用户已确定主题
- 重点是提高成稿效率

### 3. 已有初稿，进入修订

`文昌路由.skill`
-> `文昌·诊文`
-> `文昌·整章`
-> `文昌·封面`
-> `文昌·出刊`

适用：

- 用户已经写了一版
- 需要从“能读”变成“能发”

### 4. 只做视觉补全

`文昌路由.skill`
-> `文昌·配图`
-> `文昌·封面`

适用：

- 正文已定
- 需要补插图和封面方案

## 用户使用方式

理想情况下，用户不需要记忆所有子技能名，只需要记住：

- 总品牌：`文昌.skill`
- 主入口：`文昌路由.skill`

也就是说，默认应该优先让用户对总调度说需求，而不是强迫用户手动选模块。

推荐的用户表达方式：

- `帮我找最近适合公众号写的 5 个 AI 选题`
- `把这个题做成一篇知乎文章`
- `这篇稿子帮我诊断一下`
- `把这篇改成适合小红书发布的版本`
- `补一下封面标题和配图方案`
- `直接给我最终可发布版本`

只有在用户明确知道自己要哪个环节时，才直接调用子技能，例如：

- `用 文昌·定题 帮我筛这 8 个选题`
- `用 文昌·诊文 看看这篇稿子的问题`

## README 里应该明确写什么

如果后面要把这套技能真正做成仓库 README，建议至少包含这些内容：

1. `文昌.skill` 是什么
2. 为什么这套系统必须有 `文昌路由.skill`
3. 每个子技能的具体作用、输入、输出和边界
4. 常见调用链路
5. 用户应该优先如何使用
6. 哪些判断应该交给用户保留

这里最重要的不是“把技能列出来”，而是让用户一眼明白：

- 该从哪个入口进
- 每个技能解决什么问题
- 它们之间怎么配合
